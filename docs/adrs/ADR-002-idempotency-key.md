# ADR-002: Idempotency key derivation

* **Status:** accepted
* **Date:** 2026-01-08

## Context

Every workflow step needs a stable identity so a retry, a duplicate
client request, or a worker process that picks up the same `run_id`
twice ends up reading from cache instead of re-running the agent. The
key has to be:

* deterministic from `(run_id, agent, question)`,
* sensitive to a change in any one of those three components,
* short enough to be cheap to log and store as a Redis key,
* unambiguous in a shared keyspace (no collision with `run:` keys).

## Options considered

* **UUID per request** — random, so it can't be regenerated by a
  retry. Defeats the purpose.
* **Plain SHA-256** — fine, but 64-char hex keys bloat logs and don't
  add protection over what we'd actually need.
* **Plain string concatenation** — readable, but `run_id|agent|question`
  is unbounded user input that could overflow log lines or hit Redis
  key-size limits.
* **`blake2b(..., digest_size=12)` hex digest, prefixed with `step:`**
  — 96 bits is well above any realistic collision threshold for
  in-flight runs, the digest is deterministic, and `blake2b` is fast
  enough that we don't have to cache the key.

## Decision

```python
dedup_key = "step:" + blake2b(f"{run_id}|{agent}|{question}", digest_size=12).hexdigest()
```

The `step:` namespace prefix means the idempotency cache and
`run:`/`run:{id}:steps` keys share a Redis instance without collision.

## Consequences

* **Good:** keys are 29 chars, fixed-width, log-safe.
* **Good:** Hypothesis property tests
  (`tests/test_dedup_property.py`) lock the contract: same triple
  ⇒ same key, any one differing component ⇒ different key, format
  always `step:{24-hex}`.
* **Trade-off:** the user-visible question text becomes part of the
  key. Two questions that *should* canonicalise the same way (extra
  whitespace, case) won't share a cache entry. We accept this — agent
  results are cheap enough that conservative-side errors here are
  fine, and a user noticing two near-identical questions is more
  important to surface than to silently dedupe.
